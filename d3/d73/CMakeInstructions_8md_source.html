5 > Note that this document describe the [new CMake system](CMakeMigration.md) : the one based on buckets has been discontinued.

7 ## Instructions for contributors (aka developers' documentation)
8 
9 A sub-module `CMakeLists.txt` defines one or more _targets_.
10 A target generally corresponds to an actual build artifact like a (static or shared) library or an executable. Targets are the cornerstone of any modern cmake build system.
11 
12 ## Typical CMakeLists.txt
13 
14 A typical module's `CMakeLists.txt` contains
15 
16 - a call to [o2_add_library](../cmake/O2AddLibrary.cmake) to define a library (and its dependencies)
17 - call(s) to [o2_add_executable](../cmake/O2AddExecutable.cmake) to define one or more executables (and their dependencies)
18 - call(s) to [o2_add_test](../cmake/O2AddTest.cmake) to define one or more tests (and their dependencies)
19 
20 Optionally it might contain a call to [o2_target_root_dictionary](../cmake/O2TargetRootDictionary.cmake) if the module's library requires a Root dictionary.
21 
22 All _direct_ dependencies must be _explicitely_ defined with the `PUBLIC_LINK_LIBRARIES` keyword of the various o2_xxx functions.
23 
24 Note that despite the parameter name, the `PUBLIC_LINK_LIBRARIES` should refer to _target_ names, not library names. You _have to_ use the fully qualified `O2::targetName` and not the short `basename` you might have used to _create_ the target. Note also that if the referenced target does not exist, CMake will tell you right away at the configure stage (which is a good thing).
25 
26 Note also CMakeLists.txt should be considered as code and so the same care you put into writing code (e.g. do not repeat yourself, comments, etc...) should be applied to CMakeLists.txt. Also, like the rest of our code, we can take of the formatting using the [cmake-format](https://github.com/cheshirekow/cmake_format) tool (that tool is certainly not as robust as `clang-format` but it can get most of the job done easily).
27

83 ### [Ex2](../Examples/Ex2) Adding a basic library with a Root dictionary
84 
85 Using a slightly modified version of the previous example (the [A.h](../Examples/Ex2/include/Ex2/A.h) now uses ClassDef for instance), we'll now add a Root dictionary :
86 
87  Ex2
88  ├── CMakeLists.txt
89  ├── include
90  │   └── Ex2
91  │   └── A.h
92  └── src
93  ├── A.cxx
94  ├── B.cxx
95  ├── B.h
96  └── Ex2LinkDef.h
97 
98  o2_add_library(Ex2 SOURCES src/A.cxx src/B.cxx PUBLIC_LINK_LIBRARIES FairMQ::FairMQ)
99  o2_target_root_dictionary(Ex2 HEADERS include/Ex2/A.h src/B.h LINKDEF src/Ex2LinkDef.h)
100 
101 will define a library, that depends on the `FairMQ::FairMQ` target, with 3 source files (the provided `A.cxx` and `B.cxx` plus a dictionary added by the `o2_target_root_dictionary`). In addition to create a dictionary source file, the `o2_target_root_dictionary` function also appends a dependency on `ROOT::RIO` target to `Ex2` because it's needed at link time for the dictionary part.
102 
103 While the `HEADERS` parameter to `o2_target_root_dictionary` is mandatory, the `LINKDEF` one can be omitted if the LinkDef file is named \[targetBaseName]LinkDef.h and is located in the \[targetBaseName] source directory (so in the above example it could have been omitted).
104 
105 When doing a `cmake --build .` you'll now find, in addition to the library in the `stage/lib` dir, a `rootmap` file and a `pcm` file. Those two files must be collocated with the library if you want to be able to load that library easily from the Root prompt.

109  stage/lib/libO2Ex2.dylib

141 ### [Ex3](../Examples/Ex3) Adding an executable
142 
143 Adding an executable to previous example :
144 
145  Ex3
146  ├── CMakeLists.txt
147  ├── include
148  │   └── Ex3
149  │   └── A.h
150  └── src
151  ├── A.cxx
152  ├── B.cxx
153  ├── B.h
154  ├── Ex2LinkDef.h
155  └── run.cxx
156 
157  o2_add_library(Ex3
158  SOURCES src/A.cxx src/B.cxx
159  PUBLIC_LINK_LIBRARIES FairMQ::FairMQ)
160 
161  o2_target_root_dictionary(Ex3
162  HEADERS include/Ex3/A.h src/B.h)
163 
164  o2_add_executable(ex3
165  SOURCES src/run.cxx
166  PUBLIC_LINK_LIBRARIES O2::Ex3 O2::Ex2
167  COMPONENT_NAME example)
168 
169 There are three things to note in the `o2_add_executable`.
170 
171 First, we reference the `Ex3` library, built in the same module, using its fully qualified name `O2::Ex3`. Using just `Ex3` would not work, as there is no target named `Ex3` (Ex3 is just the basename of the target). Likewise, the "external" (to this module) Ex2 library is also referenced by its fully qualified name `O2::Ex2`. The target dependencies gather the link dependencies (to the relevant libraries used during linking) but also the include dependencies (so that the relevant include directories, e.g. `include/Ex2` are found when compiling).
172 
173 Second, we used the optional `COMPONENT_NAME` argument, that will be used as part of the executable name. The output executable name will be `o2-example-ex3`, following our [naming convention for executable](https://rawgit.com/AliceO2Group/CodingGuidelines/master/naming_formatting.html#Executable_Names).
174

204 ### [Ex4](../Examples/Ex4) Adding a couple of tests
205 
206 Let's add two basic test2 to our previous example.
207 
208  o2_add_test(test1
209  SOURCES test/test1.cxx
210  PUBLIC_LINK_LIBRARIES O2::Ex4
211  COMPONENT_NAME Ex4
212  LABELS fast dummy obvious
213  INSTALL)
214 
215  o2_add_test(test2
216  SOURCES test/test2.cxx
217  PUBLIC_LINK_LIBRARIES O2::Ex4 O2::Ex3 O2::Ex2
218  COMPONENT_NAME Ex4
219  LABELS fast dummy)
220

314 ### [Ex5](../Examples/Ex5) Adding a man page
315 
316 If a module provides one or more executables, it might be of interest for the users of those executables to have access to a man page for them. Ex5 illustates that use case.
317 
318  .
319  ├── CMakeLists.txt
320  ├── README.md
321  ├── doc
322  │   └── ex5.7.in
323  └── src
324  └── run.cxx
325 
326 The [man page](ManPages.md) is created using :
327 
328  o2_target_man_page([targetName] NAME ex5 SECTION 7)
329 
330 where `NAME xx` refers to a file `doc/xx.[SECTION].in`, and the actual `targetName` can be found from the base target name (ex5 in that case) using the [o2_name_target](../cmake/O2NameTarget.cmake) function.
